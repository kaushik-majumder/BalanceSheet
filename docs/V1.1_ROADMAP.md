# v1.1 roadmap — Feedback → Jira + admin web app

Captured during v1.0 launch prep so it doesn't get lost. Don't start
on any of this until v1.0 has shipped to the Play Store and been
live for at least 2–3 weeks of real user feedback. Reason: §3
(admin web app) depends on which gaps the existing Firebase + Jira
consoles actually leave you with — and you can't predict that
until users have been using the app for a bit.

> **Status legend**: ❓ open decision · 🛠 ready to build once decided · 👤 your call

---

## 1. In-app feedback that auto-creates Jira issues

User journey: Settings → "Send feedback" → pick category (Bug /
Feature request / General) → type description → optional screenshot
→ Submit → toast "Thanks, we'll look into it" → Jira issue appears
in your backlog within seconds.

### App-side work (~1 day of coding once decisions below are made)

- New screen `app/feedback.tsx` reached from Settings
- Form fields:
  - **Category** segmented control: Bug / Feature request / General
  - **Title** (single line, required, 8-80 chars)
  - **Description** (multiline, required, 20+ chars)
  - **Screenshot** (optional, via expo-image-picker, capped at 2 MB)
  - **Auto-captured metadata** (hidden from user): app version,
    OS version, device model, current user UID, current household
    ID, last 50 lines of in-memory log buffer
- Submits via `fetch` to the backend endpoint (see §1.2)
- Success / failure UI states; offline submissions queued in
  SecureStore and retried on next app foreground

### 1.2 Backend endpoint — ❓ where does this live?

Jira API tokens give write access to your project, so they MUST NOT
be bundled in the app. The backend mediates. Three realistic
options:

| Option | Pros | Cons |
|---|---|---|
| **Add a `/feedback` route to the existing Cloudflare Worker** (`scripts/parse-receipt-worker.ts`) | Free, you already deployed it, no new infra | The worker grows beyond its single-purpose name; consider renaming to a generic `api-worker` |
| **A separate Cloudflare Worker** | Cleaner separation, single-responsibility | Two deploys to maintain |
| **Firebase Cloud Functions** | Native Firestore integration — could log every submission to a `feedback/{id}` collection for moderation history before it hits Jira | Needs **Blaze plan** ($) |

**My recommendation**: extend the existing Cloudflare Worker. Free,
you've already paid the setup cost for it, and the deduplication
between this and the parser route is minimal. Rename it to
something like `api-worker` to reflect the broader scope.

### 1.3 Jira side — 👤 you need to do this once

- Sign up for a free **Atlassian Cloud** account if you don't have
  one. Free tier supports up to 10 users — plenty for personal use.
- Create a project, e.g. `RS` (Receipt Scanner). Type: Scrum or
  Kanban, your call.
- Enable two issue types under that project: **Bug** and **Story**.
  Map "Feature request" from the app to Story; "Bug" to Bug;
  "General" to Story with a `general-feedback` label.
- Create an API token at <https://id.atlassian.com/manage-profile/security/api-tokens>
  → set as the secret `JIRA_API_TOKEN` on the worker (via `wrangler
  secret put JIRA_API_TOKEN`).
- Also store `JIRA_EMAIL` (the email tied to the token) and
  `JIRA_BASE_URL` (e.g. `https://kaushikmajumder.atlassian.net`) as
  non-secret env vars in `wrangler.toml`.

### 1.4 API contract

Endpoint: `POST https://<your-worker>.workers.dev/feedback`

Headers:
- `Content-Type: application/json`
- `X-RS-Auth: <shared secret>` — same pattern as the parser route
  (`PARSE_ENDPOINT_SECRET`). Don't rely on rate-limiting alone.

Body:
```json
{
  "category": "bug" | "feature" | "general",
  "title": "string",
  "description": "string",
  "metadata": {
    "appVersion": "1.0.0",
    "osName": "Android",
    "osVersion": "14",
    "deviceModel": "Pixel 7",
    "uid": "abc123",
    "householdId": "xyz789",
    "createdAt": "2026-05-14T19:23:00Z"
  },
  "screenshotBase64": "iVBOR..." | null
}
```

Worker behaviour:
1. Verify `X-RS-Auth` header matches the shared secret
2. Validate body schema
3. Map `category` → Jira `issuetype`
4. Build the Jira issue title (use form title) and description
   (Markdown including the metadata block)
5. POST to `${JIRA_BASE_URL}/rest/api/3/issue` with Basic auth
   (`${JIRA_EMAIL}:${JIRA_API_TOKEN}`)
6. If a screenshot was sent, attach it as a second call to
   `/rest/api/3/issue/{issueKey}/attachments`
7. Return `{ ok: true, issueKey: "RS-42" }` to the app
8. Log every request to a Cloudflare KV namespace for auditing (or
   skip if you trust your own users)

### 1.5 Decisions to make before starting

| ❓ Decision | Default if undecided |
|---|---|
| Cloudflare Worker (recommended) vs Cloud Functions? | Cloudflare Worker |
| Rename existing worker to `api-worker`, or add `/feedback` route to the existing one as-is? | Add the route in-place; rename only if a third route appears |
| Show issueKey in the in-app success toast ("Created RS-42") or hide it? | Hide it — most users don't care |
| Allow anonymous feedback (no signed-in user)? | No — require sign-in |

## 2. "Email me when a bug is ready to test"

The phrasing in the original request was ambiguous. Three different
implementations depending on what you actually meant:

### 2a. Jira-status-driven email

Easiest path, no code. In Jira → Project settings → Automation:

- **Trigger**: Issue status changes to "Ready for QA" (or whatever
  status name you pick)
- **Action**: Send email to `kmaz285@gmail.com` with the issue
  fields (key, title, description, last comment, link)

Atlassian Cloud free tier includes ~100 automation runs/month.
Plenty for personal use.

**Pick this one if**: you manage Jira issues manually (or a
co-developer does), and want a notification when one moves to QA.

### 2b. CI-driven email

When a fix lands and the production build succeeds, email the
linked Jira ticket(s) found in the commit message(s) since the last
deploy.

- Extend `.github/workflows/android-build.yml`
- After the build summary step, parse git commits between the last
  tagged release and HEAD for `RS-\d+` references
- For each, fetch the issue summary via Jira API and email a digest

**Pick this one if**: you want notifications tied to actual deployable
fixes, not just status changes.

### 2c. Claude-agent-driven email

You'd set up a **scheduled Claude Code agent** that:
- Watches the Jira queue every N hours
- Picks up the next Bug labeled `ready-for-claude`
- Reads the issue, reproduces the bug locally, fixes it, opens a PR
- Emails you with PR link + before/after summary

This is **not a v1.1 app feature** — it's a separate ongoing dev
workflow. Doable today via `claude-code` CLI + cron / GitHub
Actions schedule. Document in a separate `.claude/workflows/` doc
when you're ready to set it up.

### 2.1 Decision

| ❓ Decision | Default if undecided |
|---|---|
| Which of 2a / 2b / 2c is the actual desired feature? | **2a** for v1.1; 2c later as a separate workflow |

## 3. Admin web app

Worth deferring scope until you've been live for a few weeks.
Most "admin" needs are covered by Firebase Console or Jira directly —
the question is which gaps remain.

### 3.1 What's free out of the box

| Function | Already available where |
|---|---|
| View / search all users | Firebase Console → Authentication → Users |
| View all households + receipts | Firebase Console → Firestore Data |
| Disable a user account | Firebase Console → Authentication → user → Disable |
| Reset a user's password | Firebase Console → Authentication → user → Reset |
| Triage feedback / bugs | Jira board |
| Watch app crashes | (Need to add) Firebase Crashlytics → console |
| View AI parser cost / usage | Google AI Studio dashboard, Cloudflare Worker logs |

### 3.2 What requires a custom admin app

Possible asks that don't have a free-console answer:

- **Bulk operations** — e.g. "delete all data for these 5 users"
  scripted instead of clicking through 5 console pages
- **Custom analytics dashboards** — DAU, scans per day, top
  categories across all users, conversion funnels — Firebase
  Analytics has some of this but not always in the shape you want
- **Push notification campaigns** — send a one-off broadcast or a
  targeted message to a segment
- **App-config remote-control** — change the category list, change
  pricing tiers, change feature flags, without shipping a new APK
- **Suspended-household / soft-delete management** — actions that
  span multiple Firestore collections in one transaction
- **Customer-support views** — paste a user email, see their
  household + recent receipts + crashes side-by-side

### 3.3 Decision framework

After v1.0 has been live for 2–3 weeks, look at the support tasks
you've actually had to do. If 80% of them were "look up X in
Firebase console" or "answer Y email", skip the admin app —
overhead doesn't justify the build cost. If you found yourself
repeatedly running ad-hoc Firestore queries, writing one-off
scripts, or wanting bulk actions, then it's worth the build.

### 3.4 If you decide to build it

Recommended stack (cheapest, fastest):

| Layer | Pick | Why |
|---|---|---|
| Framework | **Next.js (App Router)** | First-class TypeScript, server components, easy auth gates |
| Hosting | **Vercel free tier** | Auto-deploys from GitHub, zero infra setup |
| Auth | **Firebase Auth** (same as the mobile app, but restrict to your email) | Reuses existing identity infra |
| Data | **Firebase Admin SDK** + Firestore | Same backend, full read/write privileges |
| External APIs | **Jira REST API**, **EmailJS API** | Same ones used elsewhere |
| Styling | **Tailwind** + **shadcn/ui** | Production-ready components, no design time |

Initial scope for v1 of admin app:
- Sign-in gated to your email only (hardcoded allow-list)
- Users table with search + view-detail
- Household detail page (receipts, members, photo storage usage)
- Feedback queue (mirrors Jira backlog, no editing here — Jira is
  source of truth)
- Crash log viewer (if Crashlytics is set up by then)

Estimate: 3–5 days for the first useful version.

### 3.5 Decisions to make before starting

| ❓ Decision | Default if undecided |
|---|---|
| Build admin app at all, or stick with Firebase Console + Jira? | Wait 2–3 weeks post-launch, then re-evaluate |
| If building: Next.js + Vercel? Or different stack? | Next.js + Vercel |
| Subdomain for it? | `admin.receiptscanner.app` if you've bought the domain, else `<vercel-default>.vercel.app` |
| Multi-admin or just you? | Just you for v1; allowlist of emails for future |

## Ordering for the actual v1.1 sprint

When you come back to this, the suggested order:

1. **Set up Atlassian Cloud + Jira project** (30 min, §1.3)
2. **Pick "ready-for-test" notification path** — §2a is fast (1 hour
   in Jira UI); §2b is medium (half-day workflow change); §2c is its
   own thing
3. **Add `/feedback` route to the existing Cloudflare Worker**
   (~half day, §1.4 spec)
4. **Build `app/feedback.tsx` screen + wire to Worker** (~half day)
5. **Ship as an OTA** (no native changes, no new APK needed)
6. **Wait 2 weeks**, observe what kinds of feedback show up
7. **Re-read §3.3 decision framework**, decide whether to build the
   admin web app
8. **If yes**: 3–5 days for the first useful version

Total v1.1 effort if you build everything: ~7–10 days spread
across a couple weeks. Total if you defer the admin app: ~1.5 days.

## Out of scope for v1.1 (defer to v1.2+)

- Push notifications (need FCM tokens stored, segmentation logic,
  composer UI in admin app)
- In-app analytics dashboards visible to users
- Multi-language support
- iOS launch (App Store has its own checklist that mirrors but
  doesn't equal Play's; deserves its own pass)
- Apple Sign In (required for App Store if you support Google sign-in)
- Subscription / paid tier
